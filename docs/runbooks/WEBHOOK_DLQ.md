# WorkOS Webhook Queue and DLQ Runbook

This backend ingests WorkOS webhooks at `POST /v1/webhooks/workos`, verifies the
WorkOS signature on the raw request body, validates the event shape, then enqueues
the event onto Cloudflare Queues. The queue consumer processes events through
`src/node/queue.ts` and `src/node/lib/services/webhook-processor.ts`.

## Expected Flow

1. WorkOS calls `POST /v1/webhooks/workos`.
2. The Worker verifies `workos-signature` and enqueues the validated event.
3. The main queue consumer processes the event inside a DB scope.
4. Successful events are acked.
5. Failed events are retried by Cloudflare Queues.
6. After `max_retries = 5`, Cloudflare moves the message to the DLQ.
7. The DLQ consumer records a Sentry exception and writes a `WEBHOOK_FAILED`
   audit row before acking the dead-lettered message.

## Queue Names

Names come from `wrangler.toml` and should be generated by `pnpm init-project`:

| Environment | Main queue | Dead-letter queue |
|---|---|---|
| Local | `<project>-webhooks-local` | none |
| Staging | `<project>-webhooks-staging` | `<project>-webhooks-dlq-staging` |
| Production | `<project>-webhooks-production` | `<project>-webhooks-dlq-production` |

Create queues before the first deploy of each environment:

```bash
wrangler queues create <project>-webhooks-staging
wrangler queues create <project>-webhooks-dlq-staging
wrangler queues create <project>-webhooks-production
wrangler queues create <project>-webhooks-dlq-production
```

When adding or changing queue consumers, run one plain deploy so Cloudflare registers
the consumer binding:

```bash
pnpm deploy:staging:simple
pnpm deploy:production:simple
```

After that, normal gradual deploys are fine.

## Alerts

Set these alerts before production traffic:

- Sentry alert on exception message `Webhook permanently failed`.
- Sentry alert on repeated `Webhook processing failed; will retry` logs or related
  queue-consumer exceptions.
- Cloudflare notification for queue backlog or delivery failures, where available
  for the account plan.
- Operational check on `audit_logs` rows where `action = 'WEBHOOK_FAILED'`.

The durable compliance signal is the `WEBHOOK_FAILED` audit row. Sentry is the
human-facing pager signal.

## Triage

When an alert fires:

1. Find the event id from Sentry tags or Workers logs: `eventId`.
2. Query `audit_logs` for `WEBHOOK_FAILED` with that `resource_id`.
3. Check the event type in `metadata.eventType`.
4. Inspect recent code/config changes touching user provisioning, organization
   provisioning, DB migrations, WorkOS schemas, or queue bindings.
5. Check whether the failure is transient infrastructure, schema drift, duplicate
   idempotency state, or an unsupported WorkOS event payload.

Useful query:

```sql
select id, timestamp, resource_id, error_message, metadata
from audit_logs
where action = 'WEBHOOK_FAILED'
order by timestamp desc
limit 50;
```

## Replay

Prefer replaying from WorkOS when possible:

1. Open the WorkOS dashboard.
2. Locate the failed event id.
3. Redeliver the event to the environment webhook URL.
4. Confirm the event processes successfully and no new `WEBHOOK_FAILED` row appears.

If WorkOS replay is unavailable, manually reproduce only after capturing the exact
original event body from logs or an internal incident record. Do not invent payloads
for production repair.

## Acknowledging a Permanent Failure

A dead-lettered event is only acceptable to leave unresolved when all are true:

- The event is confirmed obsolete or duplicated.
- The corresponding user/org state is already correct.
- A `WEBHOOK_FAILED` audit row exists.
- The incident notes explain why no replay is required.

Otherwise, fix the root cause and replay.

## Common Causes

- Missing or stale `WORKOS_WEBHOOK_SECRET`.
- DB migration not applied before webhook delivery.
- WorkOS added or changed an event payload field.
- Queue consumer was added/changed but only gradual deploy was run; use one
  `pnpm deploy:<stage>:simple` to register the consumer.
- Idempotency row stuck from a prior failed attempt.

## Post-Fix Verification

Run these checks:

```bash
pnpm check
pnpm test:integration
pnpm deploy:staging
```

Then redeliver one known event and verify:

- The WorkOS delivery succeeds.
- No new `WEBHOOK_FAILED` audit row is created.
- The expected user/org state exists in Postgres.
- Sentry has no new queue-consumer exception for the event id.
